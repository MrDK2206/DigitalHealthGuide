Detailed flow
1. Load inputs

PDF files are collected from Data/ and turned into document objects.

2. Split text

Long documents are chunked with overlap so a query can match a relevant passage reliably.

3. Score context

The backend scores local chunks first and only falls back to Pinecone if needed.

4. Generate answer

Groq or OpenAI are optional and are used only after the document context has been assembled.

Good engineering choices
  • The app does not require online model access to start the site.
  • Local retrieval makes the project useful even when API keys are missing.
  • The frontend communicates with the backend through a simple text response.
  • Render deployment is straightforward because the project includes gunicorn and a Procfile.
What to tell a reviewer

This is a Flask-based retrieval chatbot for medical PDFs. It demonstrates document ingestion, local semantic matching, optional vector search, model fallback strategy, and a clean UI layered on top of the backend.

That is the strongest technical summary for interviews or portfolio presentations because it explains both the value and the implementation tradeoffs.
